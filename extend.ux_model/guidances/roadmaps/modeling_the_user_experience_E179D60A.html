Путеводитель: Modeling the User Experience
This Roadmap describes how user-experience modeling can be used throughout a phased, iterative development lifecycle.
Основное описание

Introduction

The interaction a user has with an application, the application's user experience, has a profound affect on the success of the application. Both the presentation and business teams must collaborate on the development of the user experience. The presentation team provides the look-and-feel of the application, making sure that it is consistent with the user experience of the other applications the user interacts with. The business team provides the business content that supports the business features provided by the application. The key to a successful collaboration is the development of a contract that both parties understand and agree to support. The  User-Experience Model such a contract. The User-Experience Model is represented using UML, so it integrates seamlessly with the other system models to provide a clearer picture of the entire application.  For more information, see Guideline: Modeling the User Experience.

The user-experience activities occur primarily during Inception and early Elaboration.   A brief overview of these activities is provided in the following tables.

  • Inception
  • Elaboration
  • Construction
  • Transition

For a discussion on the parallels between User-Experience Modeling and business logic modeling, see User Experience and Business Logic Process Parallels. 

RUP Inception Phase Activities

Brief Description User-Experience Modeling Activities

The user-experience activities begin early in the project lifecycle when the standards, guidelines and conventions for modeling the user-experience for the current project (i.e., the system's look and feel, emotion, etc.) are established.  These "ground rules" are documented in project-specific guidelines.

Activity: Prepare Environment for Project
  • Task: Tailor the Development Process for the Project
  • Task: Prepare Guidelines for the Project

RUP Elaboration Phase Activities

Brief Description User-Experience Modeling Activities
Analysis

Modeling the user experience occurs as part of Task: Design the User Interface. The focus is on the user-experience of the Use Cases being developed for the current iteration, identifying the  User-Experience Elements that participate to perform the Use Cases, how the elements interact to perform the Use Cases ( User-Experience Storyboards), and the navigation paths between the elements that support the interaction ( User-Experience Navigation Map).  All of these work products are part of the  User-Experience Model. 

If traceability is being maintained between the User-Experience Model and the Analysis Model, this traceability should be established.  For more information, see  Guideline: User-Experience Model.

A User-Interface Prototype may then be developed to validate the resulting user-interface design (and the User-Experience Model).

The User-Experience Model is then reviewed (Task: Review the Design).

Activity: Analyze Behavior
  • Task: Design the User Interface (Guideline: User Experience Modeling )
  • Task: Prototype the User-Interface
  • Task: Review the Design (  Checklist: User-Experience Element,  Checklist: User-Experience Element,  Checklist: User-Experience Navigation Map,  Checklist: User-Experience Storyboard, Checklist: Analysis Class, Checklist: Design Model, Checklist: Use-Case Realization)
Architecture Design

During architecture design, the presentation aspects of the system (documented in the  User-Experience Model) and the business aspects of the system (documented in the Analysis Model, if created, and Design Model), are integrated into a single overall architectural design that will be taken forward into detailed design and implementation. 

Design elements are identified from Analysis Classes and  User-Experience Element.  If traceability is being maintained between the User-Experience Model and the Design Model, this should be captured at this time.  This traceability will evolve as the design elements mature during detailed design.  For more information on User-Experience Model traceability, see  Guideline: User-Experience Model.  

The result is a single Design Model (with both presentation and business design elements), an overall software architecture (with a User-Experience View).  For an example of a user-experience view, see Example: User-Experience View.  

Activity: Refine the Architecture
  • Task: Identify Design Elements 

Detailed Design and Implementation

During detailed design and implementation, the Software Architect participates in design and code reviews to make sure that the evolving design and implementation remain consistent with the  User-Experience Model.

If traceability is being maintained between the User-Experience Model and the Design Model, this traceability should evolve as the design classes mature during detailed design.  For more information on User-Experience Model traceability, see  Guideline: User-Experience Model.

  • Task: Review the Design


RUP Construction Phase Activities

Brief Description User-Experience Modeling Activities
The Construction phase is where the majority of the implementation is produced.  The amount of user-experience modeling that is done relative the amount of implementation and testing is quite small. However, The Software Architect participates in design and code reviews to make sure that the evolving design and implementation remain consistent with the  User-Experience Model.

If traceability is being maintained between the User-Experience Model and the Design Model, this traceability should evolve as the design classes mature during detailed design.  For more information on User-Experience Model traceability, see  Guideline: User-Experience Model.

  • Task: Review the Design


RUP Transition Phase Activities

Brief Description User-Experience Modeling Activities
Transition is where the system is transitioned to the user community.  However, in a user-centered development effort, you shouldn't wait until Transition phase to involve the user.  When the user has been involved throughout the development, formal beta and acceptance testing is often significantly minimized or non-existent. Instead, detailed user feedback and approval occurs throughout the development effort.

A successful transition is dependent on the development of, and the adherence to, a successful  User-Experience Model.  Without such a model, the user community would not be prepared for what is being handed off to them and there is a risk that they may not like it.

User Experience and Business Logic Process Parallels

There are two very important system aspects, the presentation aspect (the  User-Experience Elements and the information contained on them) and the business logic aspect (the design elements that provide the business functionality and the information displayed on the User-Experience Elements).  The presentation aspect is represented by the  User-Experience Model and the business logic aspect is represented by the Analysis Model and the Design Model). These aspects are developed at the same time. 

During analysis, there are "parallel threads" for the presentation and business logic aspects of the system. 

  • Presentation Aspect. Task: Design the User Interface focuses on the user-experience of the Use Cases being developed for the current iteration, identifying the  User-Experience Elements that participate to perform the Use Cases, how the elements interact to perform the Use Cases ( User-Experience Storyboards), and the navigation paths between the elements that support the interaction ( User-Experience Navigation Map). 
  • Business Aspect. Task: Use-Case Analysis focuses on the analysis of the Use Cases being developed for the current iteration, except not much time is spent detailing the user interface.  Instead boundary Analysis Classes are identified as "holding places" for the user interface. By deferring the details of the presentation aspect to the User-Experience Model, you can concentrate on the business aspects when looking at the Analysis Classes (e.g., the interactions to perform the necessary business functionality).  This is a nice separation of concerns.  

During design, these aspects are integrated to provide one "system solution" that is carried through to implementation.
